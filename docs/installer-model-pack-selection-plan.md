# Installer Model-Pack Selection Plan

## 1. Purpose

This document audits the current v3dfy installer architecture and proposes the
implementation plan for optional model-pack selection in both the web installer
and the offline installer.

This is a design/report slice only. No installers, release payloads, model-pack
ZIPs, downloads, or commits were created.

## 2. Current Installer Architecture Summary

Current release installation is a two-layer bootstrap design:

- `scripts/publish-win-x64.ps1` publishes the self-contained Windows x64 app,
  publishes `V3dfy.SetupHelper.exe`, copies tools/licenses, and either copies or
  stages `engine\iw3`.
- `scripts/package-release-payload.ps1` creates
  `artifacts\release\v3dfy-v<version>-win-x64-portable.zip`, splits it into
  three `.partNN` files, and writes `SHA256SUMS.txt`.
- `scripts/package-release-installers.ps1` validates freshness of the portable
  ZIP and split parts, publishes a self-contained SetupHelper for installer use,
  generates `payload-manifest-v<version>.json`, and compiles both web and
  offline bootstrap installers from `packaging\inno\v3dfy-payload-bootstrap.iss`.
- `packaging\inno\v3dfy-payload-bootstrap.iss` embeds only SetupHelper and the
  payload manifest as temporary files. It does not embed the full payload.
- SetupHelper owns download, local part discovery, SHA256 verification, ZIP
  rebuild, extraction, target-directory staging, install swap, rollback, cleanup,
  and progress UI.

`packaging\inno\v3dfy.iss` is the older embedded-folder installer. It is useful
for small local test packages but is not the current release path for the full
payload.

## 3. Current Web Installer Flow

The web installer is compiled from `v3dfy-payload-bootstrap.iss` without the
`OfflineInstaller` define.

Current flow:

1. Inno shows an English informational page: "Internet required".
2. When installation begins, Inno extracts `V3dfy.SetupHelper.exe` and
   `payload-manifest.json` to `{tmp}`.
3. Inno runs SetupHelper with:
   - `--ui`
   - `--mode web`
   - `--manifest {tmp}\payload-manifest.json`
   - `--target-dir {app}`
   - `--work-dir {tmp}\v3dfy-payload-work`
   - `--log {tmp}\v3dfy-setup-helper.log`
4. SetupHelper reads `PayloadManifest.Parts`. The web download list is defined
   in `payload-manifest-v<version>.json`, generated by
   `scripts/package-release-installers.ps1` in `New-PayloadManifest`.
5. SetupHelper downloads each payload part from the manifest URL or from
   `releaseBaseUrl/fileName`, verifies each part's size and SHA256, rebuilds the
   portable ZIP, verifies the final ZIP SHA256, extracts to staging, verifies
   required installed paths, and swaps the staged directory into `{app}`.

## 4. Current Offline Installer Flow

The offline installer is compiled from the same Inno script with the
`OfflineInstaller` define.

Current flow:

1. Inno shows an English informational page: "Offline payload required".
2. User must keep the `.part01`, `.part02`, and `.part03` payload files beside
   the offline setup EXE.
3. Inno runs SetupHelper with the same payload arguments as web mode plus:
   - `--mode offline`
   - `--parts-dir {src}`
4. SetupHelper locates the expected part filenames in `{src}`, verifies size and
   SHA256 from the embedded manifest, rebuilds and verifies the portable ZIP,
   then installs the payload exactly like web mode.

The offline installer currently does not include or discover model-pack ZIPs.
It only finds the three main app payload parts.

## 5. Current SetupHelper Model-Pack Import Capabilities

SetupHelper already has a separate model-pack install mode:

```text
V3dfy.SetupHelper.exe model-pack install --zip <zip> --target-root <pretrained_models> --staging-root <staging> --current-iw3-version <version> [--current-v3dfy-version <version>] [--result <json>] [--log <json>]
```

This path is implemented by `SetupHelperProgram.RunModelPackCommandAsync` and
`ModelPackInstallExecutor`.

Existing strengths:

- Validates `MODEL_PACK.json` schema version, `targetRoot`, required fields,
  model entries, file entries, license references, compatible iw3 versions, and
  protected runtime dependency exclusion.
- Validates ZIP paths to prevent rooted paths, traversal, `engine/iw3` prefixes,
  `pretrained_models` prefixes, duplicates, and undeclared ZIP entries.
- Validates declared file size and SHA256 by reading ZIP entries.
- Plans same-hash already-installed files as skipped.
- Detects target conflicts when an installed file exists with different content.
- Stages files before copying to the target.
- Verifies staged files and installed files after copy.
- Rolls back files copied for the failing model pack.
- Writes machine-readable `ModelPackInstallResult`.

Current gap:

- The payload installer does not yet call model-pack import as part of setup.
- Model-pack import progress is not yet integrated into SetupHelper's global
  setup progress UI.
- There is no installer-facing model-pack release manifest.

## 7. Web Installer Proposed UX

Recommended implementation: add a model-pack selection page inside the existing
SetupHelper UI before any payload download or installation begins.

Why SetupHelper, not Inno Pascal:

- SetupHelper already owns downloads, progress, hashing, logs, and elevated
  target install.
- SetupHelper can parse JSON, render a real table, use WinForms tri-state
  checkbox behavior, and reuse model-pack validation code.
- Inno has no comfortable built-in JSON/ZIP/MODEL_PACK validation path.
- Avoids hardcoded generated Pascal arrays and fragile custom Inno table code.

Web flow with model packs:

1. Inno keeps the current requirements page, with text updated to mention that
   optional models can be selected next.
2. Inno extracts a generated model-pack release manifest alongside the payload
   manifest.
3. Inno passes the model-pack manifest path to SetupHelper.
4. SetupHelper opens its setup window and first shows "Optional model packs".
5. User may select none, one, several, or all visible model packs.
6. User clicks Continue.
7. SetupHelper downloads and installs the app payload.
8. SetupHelper downloads selected model-pack ZIPs from release assets, verifies
   outer ZIP size/SHA256, validates `MODEL_PACK.json` and inner file hashes,
   imports selected packs to the installed app, and reports results in the setup
   log.

Recommended default selection: none. The base app already includes a usable
model, and some optional packs are large. The page should make clear that users
can import packs later from the app.

Suggested page text:

```text
Optional model packs

v3dfy already includes a base model. Select optional model packs to download
and install now, or leave everything unchecked and import models later from the
app.
```

## 8. Offline Installer Proposed UX

Offline flow with model packs:

1. Inno keeps the current offline requirements page, with text updated to say
   optional model-pack ZIPs can be placed beside the installer.
2. Inno passes `{src}` to SetupHelper for payload parts and offline model-pack
   discovery.
3. SetupHelper reads the embedded model-pack release manifest and checks `{src}`
   for matching model-pack asset filenames.
4. Only detected local model-pack ZIPs from the published manifest are listed.
5. If none are detected, either skip the selection page or show a concise
   "No optional model packs found" state with Continue.
6. Selected local ZIPs are verified and imported after the app payload installs.
7. No model-pack download is attempted in offline mode.

Recommended offline no-pack text:

```text
No optional model packs found

The offline installer did not find v3dfy model-pack ZIPs beside this setup
file. v3dfy will still install with its base model. You can import model packs
later from the app.
```

Unknown ZIPs beside the installer should not be listed as official installer
options. They can be mentioned generically as "Import other model packs later
from the app." This avoids exposing unsupported, gated, blocked, or
non-published packs.

## 9. Single Top Checkbox Behavior

Do not add separate "Select all" and "Deselect all" buttons.

Use one top tri-state checkbox above the visible model list:

- Unchecked: no visible packs selected.
- Checked: all visible packs selected.
- Indeterminate: some visible packs selected.

Behavior:

- Clicking unchecked selects all visible packs.
- Clicking checked clears all visible packs.
- Clicking indeterminate selects all visible packs. This matches the practical
  GitHub Desktop-style top-checkbox behavior.
- Changing an individual row updates the top checkbox state immediately.
- If no rows are visible, the top checkbox is disabled.

The top checkbox state must only consider visible installer-eligible packs. It
must not count gated, blocked, unpublished, or absent offline packs.

## 10. Progress Model for Web Installer

Progress should stay in SetupHelper and include:

- app payload part downloads;
- app payload part verification;
- payload ZIP rebuild;
- payload ZIP verification;
- payload extraction/install;
- selected model-pack downloads;
- selected model-pack outer ZIP verification;
- selected model-pack manifest/hash validation;
- selected model-pack import/copy;
- cleanup/finalization.

Recommended implementation: replace or extend the fixed phase-range
`SetupOverallProgressTracker` with a work-plan model:

```text
SetupWorkPlan
  WorkItem: app payload download/verify/rebuild/extract/install
  WorkItem: model pack download/verify/validate/import for each selected pack
```

Each item should have a weight:

- byte-based weight when size is known;
- small fixed weight for metadata/discovery/finalization steps.

For zero selected packs, the overall progress should behave like today's app
payload progress.

For selected packs, total units must grow based on selected model ZIP sizes and
import work. A 1.34 GB model pack should visibly affect global progress more
than a 99 MB pack.

## 11. Progress Model for Offline Installer

Offline progress should include:

- finding and verifying app payload parts;
- app payload ZIP rebuild/verify/extract/install;
- selected local model-pack outer ZIP verification;
- selected model-pack manifest/hash validation;
- selected model-pack import/copy;
- cleanup/finalization.

Offline mode must not include download phases for model packs. It should show
"Verifying local model pack" and "Installing model pack" instead.

With zero selected packs, current offline progress remains effectively the same.

## 12. Hash/Validation/Security Plan

Use a generated installer model-pack release manifest embedded in the installer.

Recommended manifest fields:

```json
{
  "schemaVersion": 1,
  "v3dfyVersion": "0.1.0-preview.1",
  "modelPackReleaseBaseUrl": "https://github.com/Henrriegel/v3dfy/releases/download/v0.1.0-preview.1",
  "currentIw3Version": "nunif-d23721f1",
  "packs": [
    {
      "packId": "depth-anything-v2-small",
      "displayName": "Depth Anything V2 Small",
      "bestUseEnglish": "Good first optional model for faster general conversions.",
      "bestUseSpanish": "Buen primer modelo opcional para conversiones generales mas rapidas.",
      "assetFileName": "v3dfy-modelpack-depth-anything-v2-small-v0.1.0-preview.1.zip",
      "relativeArtifactPath": "depth-anything-v2-small/v3dfy-modelpack-depth-anything-v2-small-v0.1.0-preview.1.zip",
      "url": "https://github.com/Henrriegel/v3dfy/releases/download/v0.1.0-preview.1/v3dfy-modelpack-depth-anything-v2-small-v0.1.0-preview.1.zip",
      "zipSha256": "d45872e2d301cfc36a0a1815f55ee5bcfbdb5fd3e2d281c55b0af907e028e84c",
      "zipSizeBytes": 99253365,
      "iw3DepthModelNames": ["Any_V2_S"],
      "mappingKeys": ["depth-anything-v2-small"],
      "installerSelectable": true,
      "defaultSelected": false
    }
  ]
}
```

Validation layers:

1. Packaging script validates each selected release model-pack ZIP exists,
   matches expected size/SHA256, and is in the generated manifest.
2. Web SetupHelper downloads only manifest-listed URLs/assets.
3. Web SetupHelper verifies outer model-pack ZIP size/SHA256 before opening it.
4. Offline SetupHelper lists only local ZIP filenames present in the embedded
   manifest and verifies outer ZIP size/SHA256 before opening it.
5. `ModelPackInstallPlanner` validates `MODEL_PACK.json`, archive safety,
   declared file hashes, model/file consistency, iw3 compatibility, and
   protected runtime dependency exclusion.
6. `ModelPackInstallExecutor` verifies staged and installed files.

Inno should not validate `MODEL_PACK.json` directly. It is the wrong layer for
ZIP parsing, JSON validation, hashing large inner files, and model-pack path
rules.

## 13. Install/Import Target Path Plan

Do not hardcode installation paths.

SetupHelper should derive the model-pack target root from the selected install
directory after the app payload has been installed:

```text
{app}\engine\iw3\nunif\iw3\pretrained_models
```

Preferred implementation:

- Reuse `InternalToolPathResolver` or shared path contract logic from the app
  side.
- Use the installed target directory passed to SetupHelper as `--target-dir`.
- Use a staging root under SetupHelper's temp/work directory, not under
  Program Files.
- Read bundled iw3 version from installed `engine\iw3\IW3_CLI_CAPABILITIES.json`
  when present; fall back to the installer model-pack manifest's
  `currentIw3Version`.

The current Inno installer already requires admin privileges, so SetupHelper
runs in the elevated setup context. No separate model-pack UAC prompt should be
needed during installer-driven import.

## 14. Rollback/Error Handling Plan

App payload install remains the critical operation. If it fails, the installer
fails as it does today.

Optional model-pack failures should not uninstall or roll back the app payload.
Recommended behavior:

- Import selected model packs sequentially.
- If one pack fails validation/import, record the failure, continue with the
  remaining selected packs, and show "Installed v3dfy with model-pack warnings."
- Return success to Inno if the app payload installed successfully, even if one
  optional pack failed. The SetupHelper window and log must clearly list failed
  packs and tell the user they can import models later from the app.
- Within a single pack, keep using `ModelPackInstallExecutor` rollback for files
  copied by that pack.
- Already-installed same-hash files are skipped and counted as success.
- Existing files with different content are conflicts. Do not overwrite them.
  Fail that pack and continue with others.

If the user cancels during model-pack processing:

- Cancel current work.
- Roll back files copied by the active pack.
- Leave the app payload installed if it already succeeded.
- Show a clear cancellation summary.

  - top tri-state checkbox;
  - row checkboxes;

  - discovering model packs;
  - downloading model pack;
  - verifying model pack;

## 21. Recommended Implementation Slices

### P4B - Installer Model-Pack Manifest

- Generate `v3dfy-model-packs-v<version>.json` from model-pack artifacts.
- Include best-use text, size, SHA256, asset filename, URLs, and selectable
  flags.
- Add packaging tests.
- Do not change Inno or SetupHelper behavior yet.

### P4C - SetupHelper Selection UI and Discovery

- Add SetupHelper manifest parsing.
- Add web/offline selectable row discovery.
- Add WinForms selection page with top tri-state checkbox.
- No downloads/imports yet; just return selected pack IDs internally and test
  the UI/state logic.

### P4D - Web Model-Pack Download and Verification

- Add selected model-pack download support.
- Verify outer ZIP size/SHA256.
- Integrate model-pack download progress with global setup progress.
- Keep install/import disabled or behind test seam until P4E.

### P4E - Installer-Driven Model-Pack Import

- After app payload install, derive installed pretrained_models target root.
- Import selected verified packs using `ModelPackInstallExecutor`.
- Implement per-pack failure handling and success-with-warnings summary.
- Test already-installed, conflict, rollback, and multi-pack behavior.

### P4F - Inno and Packaging Integration

- Embed model-pack manifest into web/offline installers.
- Pass new SetupHelper arguments.
- Update installer README files.
- Add source tests for Inno/script wiring.

### P4G - Documentation and Manual Validation

- Update README and release notes template.
- Build installers in a later approved slice.
- Manually validate web and offline flows with zero, one, and multiple packs.

## 22. Risks and Open Questions

- SetupHelper UI is currently English-only. The app supports English/Spanish,
  but installer localization is not implemented. Recommendation: keep P4
  installer model selection English-only for first implementation, then add
  installer localization as a separate slice if needed.
- If the helper window appears after the user clicks Install in Inno, it is
  technically not an Inno wizard page. It still satisfies "before download/install
  starts" because SetupHelper shows selection before doing work. If a strict Inno
  wizard page is required, expect substantially more Pascal code and weaker data
  validation ergonomics.
- Large model packs can add several GB of downloads. The selection UI should
  default to none selected and show total selected download size.
- Current payload reinstall replaces the whole `{app}` tree, so previously
  imported model packs are removed by app payload install before selected packs
  are imported. That is acceptable if the installer then imports selected packs,
  but users upgrading without selecting packs may lose previously imported packs.
  This is a product decision. Safer future work: preserve existing
  `pretrained_models` or warn during upgrades.
- The generated model-pack manifest must be tied to the exact release assets.
  Do not download a mutable model-pack catalog at install time unless the catalog
  itself is hash-pinned by an embedded manifest.
- Offline discovery by filename against the embedded manifest avoids exposing
  unknown packs but will not list valid third-party packs. Users can import those
  later from the app.

## 23. Final Recommendation

Implement installer model-pack selection in SetupHelper, not Inno Pascal.

Use an embedded generated model-pack release manifest for both web and offline
installers. Web mode lists all installer-selectable published packs from that
manifest. Offline mode lists only matching local ZIPs found beside the installer.
Use a single tri-state top checkbox and default to no optional packs selected.

Keep all security decisions in SetupHelper and the existing model-pack
planner/executor: outer release asset hash verification first, then
`MODEL_PACK.json` and inner file validation, then staged import with rollback.

Split implementation into the P4B-P4G slices above rather than doing one large
installer change.
